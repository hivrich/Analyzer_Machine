# Investigate Workflow

Это главный рабочий путь проекта.

## Как этим пользоваться

Пишешь обычный запрос:

```bash
python -m app.cli investigate <client> --query "Разберись, почему просела органика в январе 2025"
```

Или запрос про заявки:

```bash
python -m app.cli investigate <client> --query "Посмотри, почему упали заявки в феврале 2026" --refresh
```

## Что делает система

`investigate` сам:

1. понимает, про что запрос: трафик, SEO, страницы, конверсии, индексация
2. определяет периоды из запроса или берёт дефолтное сравнение
3. проверяет, какие источники доступны для клиента
4. при необходимости подбирает основную цель Метрики
5. запускает первый набор анализов
6. строит первые гипотезы
7. если данных не хватает, запускает следующий раунд и добирает только нужные данные
8. останавливается, когда причина уже достаточно понятна или новых полезных шагов больше нет
9. пишет итоговый отчёт

## Как работает цикл расследования

`investigate` работает не одним проходом, а раундами.

Обычно это выглядит так:

1. сначала общий срез по трафику и страницам
2. потом проверка, есть ли SEO-сигнал, конверсионный сигнал или сигнал по индексации
3. потом добор данных именно под эти гипотезы
4. потом повторная оценка: причина уже понятна или нужно идти глубже

Пример для SEO:

- раунд 1: Метрика по источникам и страницам
- раунд 2: страницы из поискового трафика, GSC-запросы, Вебмастер
- раунд 3: GSC-страницы, если нужно локализовать просадку точнее

Пример для заявок:

- раунд 1: трафик, страницы, конверсии по источникам
- раунд 2: конверсии по страницам, если видно падение заявок или CR

## Какие источники использует

По умолчанию:

- Метрика: всегда, если настроена
- GSC: когда запрос похож на SEO / органику / поисковые запросы / позиции
- Яндекс.Вебмастер: когда запрос похож на SEO / индексацию / органику

Если источник не настроен, расследование не падает целиком. В отчёте просто будет отмечено, что именно было недоступно.

## Что сохраняется

После каждого запуска создаётся папка:

`reports/<client>/<run_id>/`

Внутри:

- `report.md`
- `report.html`
- `evidence.json`
- `evidence.txt`

## Как подбирается конверсионная цель

Если в `config.yaml` уже есть `metrika.goal_id`, берётся он.

Если нет, система пытается выбрать цель автоматически:

1. `e_purchase`
2. цели про заявку / форму / contact_data / contact_data_sent / action
3. цели про регистрацию / заказ / checkout

Если уверенно выбрать нельзя, это прямо пишется в отчёте.

## SEO skill

Для SEO-расследований добавлен локальный skill:

`~/.codex/skills/seo-investigation/SKILL.md`

Он задаёт правила:

- когда обязательно подключать GSC и Яндекс.Вебмастер
- какие SEO-гипотезы проверять в первую очередь
- когда уходить в deep dive по страницам, запросам и индексации
- как строить итоговый SEO-отчёт

Проект не зависит только от этого skill: основные правила расследования уже встроены в `app/orchestrator/`.
