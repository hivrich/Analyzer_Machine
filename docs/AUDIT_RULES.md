# Analyzer Machine — Audit Rules

Этот документ описывает текущие, реально доступные правила и инструменты аудита.

## Цель

Перед публикацией любой аналитики проверить:

- что цифры взяты из реальных артефактов
- что расчеты воспроизводимы
- что гипотезы не выдаются за доказанные факты

## Базовые правила

### 1. Каждая цифра должна иметь источник

Допустимые источники:

- workbook в `data_cache/<client>/`
- raw / normalized JSON в `data_cache/<client>/`
- CLI output
- подготовленный отчет в `reports/`

Если источника нет, цифру использовать нельзя.

### 2. Расчеты делает код

Нельзя считать вручную:

- percent change
- contribution
- totals
- rankings

Модель только интерпретирует уже посчитанное.

### 3. Гипотеза не равна доказанному выводу

Для каждого объяснения желательно отделять:

- факт
- наиболее вероятную интерпретацию
- альтернативные причины
- что нужно проверить дальше

### 4. Audit helpers не заменяют здравый смысл

Даже если helper прошел, итоговый отчет все равно нужно проверять глазами:

- даты совпадают
- периоды не перепутаны
- основной драйвер не противоречит totals
- confidence не завышен

## Реально существующие audit-команды

### Проверка наличия и свежести данных

```bash
python -m app.cli audit-data <client> <period_start> <period_end>
```

Команда проверяет базовую доступность директорий и наличие workbook-файлов за период.

### Проверка отдельной цифры

```bash
python -m app.cli audit-metric "metric=value" --source <path-or-file> --client <client> [--period "..."]
```

Подходит для быстрой проверки конкретного утверждения против JSON-файла.

### Проверка отчета перед публикацией

```bash
python -m app.cli audit-report <report_path> [--strict]
```

Важно:

- это пока partial helper
- он не парсит отчет полностью
- его задача сейчас: напомнить обязательный checklist перед публикацией

## Рекомендуемый чеклист перед ответом пользователю

1. Файл evidence существует.
2. Периоды в evidence совпадают с запросом пользователя.
3. Цифра взята именно из этого evidence, а не пересказана "по памяти".
4. Growth/drop drivers не противоречат totals.
5. Любая гипотеза обозначена как гипотеза, а не как установленный факт.
6. Если данных не хватает, это явно сказано.

## Минимальный safe format для выводов

Рекомендуемый стиль:

1. Факт с файлом-источником.
2. Интерпретация факта.
3. Ограничение или альтернативное объяснение.
4. Следующее действие для проверки.

## Что считать красным флагом

- цифра без файла-источника
- "очевидная причина" без данных
- отсутствие периода в ссылке на evidence
- уверенное утверждение при частичных данных
- подмена capability missing догадкой
